# Deployment

This repository uses GitHub Actions for releases and NPM publishing, but they are handled by different workflows.

## Workflows

### Release workflow

Defined in `.github/workflows/release.yml`.

- Trigger: push of a Git tag matching `v*.*.*`
- Effect: creates a GitHub Release for that tag
- Release notes: generated by `scripts/generate-release-notes.js`
- Does not publish to NPM

Examples of tags that trigger this workflow:

- `v1.0.0`
- `v4.1.1`

Examples of tags that do not match this workflow trigger:

- `1.0.0`
- `release-1.0.0`
- `v1.0`

### Publish workflow

Defined in `.github/workflows/publish.yml`.

- Trigger: manual `workflow_dispatch` only
- Effect: installs dependencies, builds the package, runs `npm run test:ci`, and runs `npm publish`
- Authentication: requires the `NPM_TOKEN` repository secret

Important: pushing a Git tag does not trigger NPM publishing in the current setup.

The `publish.yml` workflow declares an optional `version` input, but the workflow does not currently use that value anywhere. Publishing therefore uses whatever version is already present in `package.json`.

### CI workflow

Defined in `.github/workflows/ci.yml`.

- Trigger: pushes to `main` or `master`, pull requests targeting `main` or `master`, or manual dispatch
- Effect: runs the test matrix and benchmark job

Important: tag pushes do not trigger this CI workflow.

## Recommended release flow

1. Update the package version in `package.json`.
2. Commit and push the version change to `main` or `master`.
3. Create and push a matching `vX.Y.Z` tag.
4. Confirm that the Release workflow creates the GitHub Release.
5. Manually run the Publish to NPM workflow from the Actions tab.

Keeping the Git tag and `package.json` version aligned avoids releasing one version on GitHub and publishing a different version to NPM.

## How to tag a release

Create an annotated tag locally and push it:

```bash
git tag -a v4.1.1 -m "Release v4.1.1"
git push origin v4.1.1
```

If you also need to push the branch commit first:

```bash
git push origin main
git tag -a v4.1.1 -m "Release v4.1.1"
git push origin v4.1.1
```

Replace `main` with `master` if that is the active default branch.

## Tag behavior summary

- Push `vX.Y.Z` tag: creates a GitHub Release
- Push non-matching tag: no release workflow runs
- Push branch commit to `main` or `master`: CI runs
- Push tag of any kind: CI does not run
- Run Publish to NPM workflow manually: package is built, tested, and published to NPM

## Manual publish checklist

Before manually running the publish workflow:

1. Verify `package.json` contains the intended version.
2. Verify the release tag matches that same version.
3. Verify the GitHub Release has been created successfully.
4. Verify the repository secret `NPM_TOKEN` is configured.
5. Start the `Publish to NPM` workflow from the Actions tab.
